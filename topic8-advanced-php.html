Topic 8: Advanced PHP

0% Complete
Analogy: If basic PHP is learning to cook simple meals (boil water, make toast), then advanced PHP is learning to manage an entire kitchen — handling multiple chefs (sessions), checking ingredients for freshness (filters), managing the fridge (cookies), dealing with power outages (exceptions), and accepting food deliveries (file uploads).

1. PHP Superglobals

What are Superglobals?

Superglobals are built-in PHP variables that are always available in every script, regardless of scope. You do not need to declare them or pass them as parameters. They are always written in ALL CAPS with underscores.

Analogy: Superglobals are like the essential tools always hanging on a mechanic's belt — wrench, screwdriver, pliers. No matter which car they are working on or where they are in the garage, those tools are always within reach. You never have to go look for them.

The Nine Superglobals

SuperglobalPurpose
$GLOBALSReference to all global variables available in the current script
$_GETAssociative array of variables passed via URL query string (GET method)
$_POSTAssociative array of variables passed via HTTP POST method (form data)
$_REQUESTContains the contents of $_GET, $_POST, and $_COOKIE
$_SERVERInformation about the server and execution environment
$_SESSIONSession variables — user data that persists across multiple page loads
$_COOKIECookies sent by the browser to the server
$_FILESInformation about files uploaded via HTTP POST (file uploads)
$_ENVEnvironment variables passed to the script via the shell or other methods

How many superglobals does PHP have?

Nine: $GLOBALS, $_GET, $_POST, $_REQUEST, $_SERVER, $_SESSION, $_COOKIE, $_FILES, $_ENV.

What does $_REQUEST contain?

The combined contents of $_GET, $_POST, and $_COOKIE.

2. $_REQUEST — Form Data Collection

How $_REQUEST Works

The $_REQUEST superglobal collects data after a form submission. It aggregates data from $_GET, $_POST, and $_COOKIE into a single array. This means it works regardless of whether the form used GET or POST method.

Analogy: Imagine a reception desk at a hotel. Guests can arrive by car ($_GET), by taxi ($_POST), or by walking ($_COOKIE). The receptionist ($_REQUEST) does not care how they arrived — she collects information from all of them into one guest book.

Example: Form with $_REQUEST

<!-- HTML Form -->
<form method="post" action="process.php">
    <label>Name: <input type="text" name="username"></label>
    <label>Email: <input type="email" name="useremail"></label>
    <button type="submit">Submit</button>
</form>
<?php
// process.php — data collected via $_REQUEST
if ($_SERVER["REQUEST_METHOD"] == "POST") {
    $name  = $_REQUEST['username'];
    $email = $_REQUEST['useremail'];
    echo "Welcome, $name! Your email is $email.";
}
?>
Warning: While $_REQUEST is convenient, for security-sensitive forms it is better to explicitly use $_POST. This ensures the data was submitted via POST and not via URL parameters (which can be logged in browser history and server logs).

3. $_SERVER — Server Information

Common $_SERVER Elements

The $_SERVER superglobal contains information about the server and the current request. Here are the most commonly used elements:

ElementDescriptionExample Value
$_SERVER['PHP_SELF']The filename of the currently executing script/profile.php
$_SERVER['SERVER_NAME']The name of the server (hostname)localhost
$_SERVER['HTTP_HOST']The Host header from the current requestlocalhost:8080
$_SERVER['HTTP_REFERER']The URL of the page that linked to this pagehttps://example.com/home
$_SERVER['HTTP_USER_AGENT']The browser/OS information of the visitorMozilla/5.0 ...
$_SERVER['SCRIPT_NAME']The path of the current script/folder/page.php

Example: Display Server Info

<?php
echo "Current page: " . $_SERVER['PHP_SELF'] . "<br>";
echo "Server name: " . $_SERVER['SERVER_NAME'] . "<br>";
echo "Your browser: " . $_SERVER['HTTP_USER_AGENT'] . "<br>";
echo "Referring page: " . ($_SERVER['HTTP_REFERER'] ?? 'None') . "<br>";
?>
Note: Some $_SERVER elements may not be set depending on the server configuration. Always check if an element exists before using it, for example with isset() or the null coalescing operator ??.

4. PHP Sessions — Persistent User Data

The Problem: HTTP is Stateless

Every HTTP request is stateless — the server treats each request as if it comes from a brand-new user. It has no memory of who you are between page loads. Sessions solve this problem.

Analogy — Stateless vs Sessions: Imagine a shop where the shopkeeper has amnesia. Every time you walk in the door, he asks "Who are you?" and "What were we talking about?" Sessions are like giving each customer a loyalty card. When you walk in, you show the card, and the shopkeeper checks his file cabinet (server) to remember everything about you — your name, what you bought last time, and what's in your shopping cart.

How Sessions Work

  1. When you call session_start(), PHP creates a unique session ID for the user
  2. This ID is stored as a cookie in the user's browser (named PHPSESSID)
  3. The actual data is stored on the server in a temporary file
  4. On each page load, the browser sends the session ID cookie back, and PHP retrieves the stored data
  5. Session variables are stored in the $_SESSION superglobal array

Session Lifecycle

Start: Call session_start() at the top of your script
Write: Store data: $_SESSION["name"] = "Ali";
Read: Access data: echo $_SESSION["name"];
Modify: Overwrite: $_SESSION["name"] = "Siti";
Destroy: session_destroy() removes all session data

Complete Session Example

<?php
session_start(); // Must be at the top, before any output

// Store session variables
$_SESSION["username"] = "Ali";
$_SESSION["role"]     = "admin";
$_SESSION["loggedin"] = true;

echo "Session started for: " . $_SESSION["username"];
?>
<?php
session_start(); // Must be called on every page that uses sessions

// Read session variables
if (isset($_SESSION["loggedin"]) && $_SESSION["loggedin"] === true) {
    echo "Welcome back, " . $_SESSION["username"] . "!";
} else {
    echo "Please log in.";
}
?>

Session Lifetime

By default, session variables last until the browser is closed. After that, they are destroyed. You can change this by setting a custom timeout.

Destroying Sessions

<?php
session_start();

// Remove all session variables
session_unset();

// Destroy the session entirely
session_destroy();
?>
session_unset() vs session_destroy():
session_unset() — removes all variables from the session, but the session itself still exists
session_destroy() — destroys the session entirely on the server (the session file is deleted)

Why is HTTP called "stateless"?

Because the server does not remember anything about the user between requests. Each request is treated independently.

What function must be called before using sessions?

session_start() — and it must be called before any HTML output.

Where is session data actually stored?

On the server (in temporary files). Only the session ID is stored as a cookie in the browser.

5. Cookies — Client-Side Storage

What is a Cookie?

A cookie is a small file that the server embeds on the user's computer via the browser. On subsequent visits, the browser sends the cookie back to the server, allowing it to recognize the user.

Analogy — Cookies vs Sessions: A cookie is like a wristband given to you at a concert. You wear it (on your computer), and whenever you walk through a door, the security guard reads it to know you have a ticket. A session is like the concert venue's internal guest list — it is stored backstage (on the server), and the wristband (session ID) is just the reference number that looks you up.

Cookies vs Sessions

FeatureCookieSession
Storage LocationUser's browser (client-side)Server
Size Limit~4 KB per cookieUnlimited (server RAM)
LifetimeCan be set to expire at a specific timeUntil browser closes (by default)
SecurityLess secure (visible to user)More secure (data stays on server)
Can be disabled?Yes, user can disable cookiesNo (requires cookies for session ID)

Cookie Limits & Rules

  • Max 20 cookies per domain
  • Max 4 KB per cookie
  • Max 300 cookies total across all domains
  • "One refresh delay" — a cookie set on one page is NOT available until the next page load. When you set a cookie with setcookie(), it is sent to the browser as a header. The browser will only send it back on the next request.
Important — "One refresh delay": If you set a cookie and immediately try to read it in the same script, it will NOT be available yet. You must wait for the next page load or refresh. This is because the cookie has been sent to the browser but the browser has not yet sent it back.

Creating a Cookie

<?php
// setcookie(name, value, expire, path, domain, security)
// Must be called BEFORE any HTML output (before <html> tag)

setcookie(
    "user",
    "Ali",
    time() + (86400 * 30),  // Expires in 30 days
    "/",
    "",
    false,  // secure (HTTPS only)
    true    // httpOnly (no JavaScript access)
);

echo "Cookie has been set!";
?>

Reading a Cookie

<?php
// Read from $_COOKIE superglobal
if (isset($_COOKIE["user"])) {
    echo "Welcome back, " . $_COOKIE["user"] . "!";
} else {
    echo "No cookie found. Welcome, new visitor!";
}
?>

Deleting a Cookie

<?php
// Delete by setting expiration to a time in the past
setcookie("user", "", time() - 3600, "/");
echo "Cookie deleted.";
?>

Where is a cookie stored — server or client?

On the client (user's browser). Session data is stored on the server.

What is the "one refresh delay"?

A cookie set on the current page is not accessible via $_COOKIE until the next page load or refresh.

How do you delete a cookie?

Call setcookie() with the same name and set the expiration time to a point in the past (e.g., time() - 3600).

6. Exception Handling

What is Exception Handling?

An exception is an error that occurs during program execution. Exception handling is a mechanism to catch and deal with these errors gracefully, instead of letting the script crash with a fatal error.

Analogy: Imagine you are driving a car. An exception is like a roadblock ahead. Without exception handling, you would crash right into it (fatal error). With exception handling, you have a try block (you start driving toward the road), a throw (someone spots the roadblock and raises an alarm), and a catch block (you take a detour around it). The program keeps running instead of crashing.

The Keywords

KeywordPurpose
tryWraps code that might cause an exception
throwTriggers (throws) an exception when something goes wrong
catchCatches the thrown exception and handles it

Basic try-catch Example

<?php
try {
    // Code that might fail
    $num = 10;
    $result = $num / 0; // Division by zero!

    if ($result === false) {
        throw new Exception("Calculation failed!");
    }

    echo "Result: $result";
} catch (Exception $e) {
    // Handle the error
    echo "Error: " . $e->getMessage();
}
?>

Without try-catch

<?php
// If an exception is thrown and not caught:
// PHP shows: Fatal error: Uncaught Exception in ...

// This is what happens without a catch block:
function divide($a, $b) {
    if ($b == 0) {
        throw new Exception("Cannot divide by zero!");
    }
    return $a / $b;
}

// This will crash the script:
echo divide(10, 0);
?>
Warning: Without a catch block, an uncaught exception causes a fatal error that stops script execution entirely. Always wrap risky code in try-catch blocks, especially database operations and file handling.

Database Connection with Exception Handling

<?php
try {
    $pdo = new PDO('mysql:host=localhost;dbname=mydb', 'root', '');
    $pdo->setAttribute(PDO::ATTR_ERRMODE, PDO::ERRMODE_EXCEPTION);
    echo "Connected!";
} catch (PDOException $e) {
    echo "Database connection failed: " . $e->getMessage();
}
?>

What happens if an exception is thrown but not caught?

PHP generates a fatal error: "Uncaught Exception" and the script stops executing.

What are the three keywords in exception handling?

try (wraps risky code), throw (triggers the exception), catch (handles the exception).

7. PHP Filters — Validate & Sanitize

Why Use Filters?

When data comes from user input (forms, URLs, uploads), you must check it for correctness and safety. PHP provides a built-in filter system for this.

Analogy — Airport Security: Validation is like checking if your passport is real and not expired — it answers "Is this document valid?" Sanitization is like the security scanner that removes any prohibited items from your bag — it answers "Let me clean this data of anything dangerous."

Two Types of Filters

TypePurposeExample
ValidateChecks if data is in the proper form. Returns true or false.Is this a valid email address?
SanitizeRemoves illegal characters from the data. Returns the cleaned string.Remove HTML tags from user input

The filter_var() Function

filter_var(variable, filter, options)

This is the primary function for filtering in PHP. It takes a variable, a filter type, and optional settings.

Validation Examples

<?php
// Validate an email
$email = "user@example.com";
if (filter_var($email, FILTER_VALIDATE_EMAIL)) {
    echo "Valid email!";
} else {
    echo "Invalid email!";
}

// Validate an IP address
$ip = "192.168.1.1";
if (filter_var($ip, FILTER_VALIDATE_IP)) {
    echo "Valid IP!";
} else {
    echo "Invalid IP!";
}

// Validate a URL
$url = "https://www.google.com";
if (filter_var($url, FILTER_VALIDATE_URL)) {
    echo "Valid URL!";
} else {
    echo "Invalid URL!";
}
?>

Sanitization Examples

<?php
// Sanitize a string — remove HTML tags
$dirty = "<script>alert('hacked')</script>";
$clean = filter_var($dirty, FILTER_SANITIZE_STRING);
echo $clean; // Outputs: alert('hacked')

// Sanitize an email
$bad_email = "user(at)example.com";
$clean_email = filter_var($bad_email, FILTER_SANITIZE_EMAIL);
echo $clean_email; // Outputs: user@example.com

// Sanitize a URL
$bad_url = "https://www.example.com/path with spaces";
$clean_url = filter_var($bad_url, FILTER_SANITIZE_URL);
echo $clean_url; // Outputs: https://www.example.com/path%20with%20spaces
?>
Tip: Always validate first, then sanitize. Validation tells you if the data is good. Sanitization cleans up data that might have illegal characters.

What is the difference between validation and sanitization?

Validation checks if data is in proper form (true/false). Sanitization removes illegal characters from data (returns cleaned data).

Which function filters variables in PHP?

filter_var($variable, $filter, $options)

8. File Uploading

How File Uploads Work

  1. User selects a file in an HTML form with type="file"
  2. Form is submitted via POST with a special encoding type
  3. PHP receives the file in the $_FILES superglobal
  4. The file is temporarily stored on the server
  5. You move it from the temp location to a permanent directory using move_uploaded_file()
Analogy: Uploading a file is like mailing a package. You (the user) put the package (file) in a mailbox (form submission). The postal service (PHP) receives it at a temporary sorting center (tmp_name). You then need to move it from the sorting center to its final destination (move_uploaded_file()). You also check the package size, type, and whether a similar package already exists before delivering it.

The $_FILES Array

When a file is uploaded, PHP stores information about it in the $_FILES superglobal:

KeyDescriptionExample
$_FILES['file']['name']Original filename on user's computerphoto.jpg
$_FILES['file']['type']MIME type of the fileimage/jpeg
$_FILES['file']['size']File size in bytes204800
$_FILES['file']['tmp_name']Temporary path on serverC:\Windows\Temp\php1234.tmp
$_FILES['file']['error']Error code (0 = no error)0

HTML Form for File Upload

<form action="upload.php" method="post" enctype="multipart/form-data">
    <label>Select a file to upload:</label>
    <input type="file" name="myfile">
    <button type="submit">Upload</button>
</form>
Required: The form must have:
• method="post" — GET cannot handle file data
• enctype="multipart/form-data" — this encoding type allows binary file data to be transmitted

PHP Upload Script

<?php
if ($_SERVER["REQUEST_METHOD"] == "POST") {

    $targetDir = "uploads/";
    $fileName  = basename($_FILES["myfile"]["name"]);
    $targetPath = $targetDir . $fileName;
    $fileType  = pathinfo($targetPath, PATHINFO_EXTENSION);

    // --- ALLOWED FILE FORMATS ---
    $allowedTypes = ["jpg", "jpeg", "png", "gif", "pdf"];

    // --- CHECK FILE FORMAT ---
    if (!in_array(strtolower($fileType), $allowedTypes)) {
        die("Error: Only JPG, JPEG, PNG, GIF, and PDF files are allowed.");
    }

    // --- CHECK FILE SIZE (max 5 MB) ---
    if ($_FILES["myfile"]["size"] > 5 * 1024 * 1024) {
        die("Error: File size must be under 5 MB.");
    }

    // --- CHECK IF FILE ALREADY EXISTS ---
    if (file_exists($targetPath)) {
        die("Error: File already exists.");
    }

    // --- MOVE FILE FROM TEMP TO TARGET ---
    if (move_uploaded_file($_FILES["myfile"]["tmp_name"], $targetPath)) {
        echo "File uploaded successfully: $fileName";
    } else {
        echo "Error: Failed to upload file.";
    }
}
?>

Security Checklist for File Uploads

  • ✅ Always check the file format (extension and/or MIME type)
  • ✅ Always check the file size
  • ✅ Always check if the file already exists to prevent overwriting
  • ✅ Generate a unique filename (e.g., using uniqid() or time) to prevent conflicts
  • ✅ Store uploads outside the web root or with .htaccess rules to prevent direct access
  • ✅ Rename files — never trust user-provided filenames

What encoding type is required for file upload forms?

enctype="multipart/form-data"

Which function moves an uploaded file to its destination?

move_uploaded_file($tmp_path, $destination)

Which superglobal holds uploaded file information?

$_FILES

Topic 8 Summary

  • Superglobals are nine built-in PHP variables always available regardless of scope (e.g., $_GET, $_POST, $_SESSION, $_FILES).
  • $_REQUEST collects data from $_GET, $_POST, and $_COOKIE.
  • $_SERVER provides information about the server environment (PHP_SELF, SERVER_NAME, HTTP_HOST, etc.).
  • Sessions solve the stateless nature of HTTP by storing user data on the server, referenced by a session ID cookie.
  • Use session_start() to begin, $_SESSION[] to read/write, session_destroy() to end.
  • Cookies are small files stored on the client's browser. Max 4KB, 20 per domain, 300 total.
  • The "one refresh delay" means a newly set cookie is not available until the next page load.
  • Exception handling uses try, throw, and catch to handle errors gracefully without crashing.
  • PHP Filters validate (check if data is proper) and sanitize (remove illegal characters) user input using filter_var().
  • File uploads require enctype="multipart/form-data", use the $_FILES superglobal, and move_uploaded_file() to save.
